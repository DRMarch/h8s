# Authelia

Authelia is the SSO / forward-authentication provider for the cluster, replacing Authentik. It is deployed at `auth.drmarchent.com` (taking over the hostname formerly used by Authentik).

## Architecture

A single ArgoCD Application (`authelia-helm`, sync-wave `-295`) deploys everything using multi-source:
- **Source 1:** Helm chart from `https://charts.authelia.com` (chart `authelia`, v0.11.6)
- **Source 2:** Values from `security/authelia/helm/values.yaml` — the single source of truth for the entire Authelia deployment
- **Source 3:** Raw manifests from `security/authelia/helm/resources/` — ExternalSecrets that pull static values from Vault (rendered by Terraform from `terraform/templates/security/authelia/`)

Secrets flow: Terraform (`terraform/vault-secrets/`) generates random values, hashes them where needed, and pushes them to Vault. ESO ExternalSecrets (templated by `terraform/templates/templates.tf`) pull from Vault into Kubernetes Secrets. The chart mounts those Secrets and Authelia reads them via `*_FILE` env vars.

```
Browser ──HTTPS──> Cilium Gateway (Envoy) ──> backend service (e.g. Harbor, Grafana)
                          │
                          ├── (per HTTPRoute filter) sub-request ──> Authelia :9091/api/authz/ext-authz/
                          │     - 200: allow, inject Remote-User/Remote-Groups headers
                          │     - 401/403: deny, redirect browser to https://auth.drmarchent.com
                          │     - The endpoint uses implementation=ExtAuthz (Envoy ext_authz protocol)
                          │       with authn_strategies CookieSession + HeaderAuthorization Basic
                          │
Browser ◀──redirect to login── Authelia (login UI at auth.drmarchent.com)
   │
   └── single admin user (file backend, password hashed at provision time)
```

## Secrets

All secrets are generated by Terraform (`terraform/vault-secrets/`) and stored in Vault. ESO ExternalSecrets pull them into Kubernetes Secrets. No Password generators, no PushSecrets — the source of truth is Vault.

| Vault path | Key(s) | K8s Secret | Consumer |
|---|---|---|---|
| `authelia/encryption-key` | `encryption-key` | `authelia-encryption-key` | Authelia storage encryption |
| `authelia/session-secret` | `session-secret` | `authelia-session-secret` | Authelia session + identity validation |
| `authelia/hmac-secret` | `hmac-secret` | `authelia-hmac-secret` | Authelia OIDC token signing |
| `authelia/admin-password` | `hash` | `authelia-admin-password` | Authelia init container (argon2 hash) |
| `authelia/grafana-oidc` | `client-secret-hash` | `authelia-grafana-client-secret` | Authelia OIDC verification (pbkdf2 hash) |
| `authelia/grafana-oidc` | `client-secret-plaintext` | `grafana-oidc-client-secret` (monitoring ns) | Grafana OIDC token exchange (plaintext) |
| `cnpg/authelia-user-credentials` | `username`, `password` | `cnpg-authelia-user-credentials` | CNPG + Authelia Postgres auth |

### Provisioning secrets (one-time bootstrap)

```bash
cd terraform/vault-secrets
terraform init
terraform apply -var='vault_token_file=/path/to/vault-init.json'
```

The admin password, CNPG password, and Grafana OIDC plaintext secret are available as Terraform outputs (`terraform output -json`).

### OIDC client secret design

Grafana's `client_secret` is stored as two keys under the same Vault path:
- `client-secret-hash` — pbkdf2 hash, consumed by Authelia to verify the secret Grafana presents
- `client-secret-plaintext` — plaintext, consumed by Grafana to present during token exchange

Authelia uses `client_secret_post` (not `client_secret_jwt`, which Grafana doesn't support — see [grafana/grafana#95168](https://github.com/grafana/grafana/issues/95168)). The pbkdf2 hash satisfies Authelia 4.39+'s deprecation of plaintext client secrets.

### Rotation

```bash
# OIDC client secret: update both keys in Vault, then restart consumers
vault kv put secret/authelia/grafana-oidc \
  client-secret-plaintext='<new-plaintext>' \
  client-secret-hash="$(docker run --rm ghcr.io/authelia/authelia:latest \
    authelia crypto hash generate pbkdf2 --password '<new-plaintext>' --no-confirm | grep '^\$pbkdf2')"
kubectl rollout restart deploy/authelia -n authelia
kubectl rollout restart deploy/grafana-deployment -n monitoring

# Admin password: update the argon2 hash in Vault, then restart Authelia
NEW_HASH=$(docker run --rm ghcr.io/authelia/authelia:latest \
  authelia crypto hash generate argon2 --password '<new-password>' --no-confirm | sed 's/^Digest: //')
vault kv put secret/authelia/admin-password hash="$NEW_HASH"
kubectl rollout restart deploy/authelia -n authelia

# OIDC HMAC: rotating invalidates all in-flight sessions
vault kv put secret/authelia/hmac-secret hmac-secret="$(openssl rand -hex 32)"
kubectl rollout restart deploy/authelia -n authelia
```

## User Management

Authelia uses a single admin user. The password hash is pre-computed at provision time (argon2 in `terraform/vault-secrets/`) and stored in Vault. At pod startup, the init container reads the pre-hashed value from the `authelia-admin-password` Secret and writes `/shared/users_database.yml`.

```bash
# Retrieve the admin password (stored in Terraform output)
cd terraform/vault-secrets
terraform output -raw authelia_admin_password
```

### Adding a new user

Users can be added by editing the init container script in `security/authelia/helm/values.yaml`. Add additional user entries under the `users:` key. Each user needs a `displayname`, `password` (hashed), `email`, and `groups` entry.

To generate a password hash for a new user:

```bash
docker run --rm ghcr.io/authelia/authelia:latest \
  authelia crypto hash generate argon2 --password "<password>"
```

Then add the user to the init container script and update the `access_control` rules in `security/authelia/helm/values.yaml` (`configMap.access_control.rules`) as needed.

## How to add a new protected service

To put a new HTTPRoute behind Authelia, edit its `http-route/<service>.yaml` and add a `type: ExternalAuth` filter. Use the Harbor or Grafana HTTPRoute as a template.

You also need a `ReferenceGrant` in the `authelia` namespace allowing the new service's HTTPRoute to backendRef the `authelia` Service. Example for a service in the `myapp` namespace:

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: myapp-external-auth-authelia
  namespace: authelia
spec:
  from:
    - group: gateway.networking.k8s.io
      kind: HTTPRoute
      namespace: myapp
  to:
    - group: ""
      kind: Service
      name: authelia
```

Since there is only the admin user, any authenticated user has access to all protected services. If you add per-service groups later, add a corresponding `access_control` rule in `security/authelia/helm/values.yaml` (`configMap.access_control.rules`).

## Access control

| Domain | Policy | Notes |
|---|---|---|
| `auth.drmarchent.com` | bypass | No auth needed to view the login page |
| `*.drmarchent.com` | one_factor | Any authenticated user (the sole admin) |

`default_redirection_url` is set to `https://argocd.drmarchent.com` so that users who visit the Authelia portal directly land on the cluster dashboard after login.

## Security context (non-root)

The cluster's `authelia` namespace is governed by the Kubernetes **Pod Security Standard `restricted`** admission policy, which rejects any pod that runs as UID/GID 0.

The chart's defaults for `pod.securityContext` are empty, so we set them explicitly in `security/authelia/helm/values.yaml:23-28`:

```yaml
securityContext:
  pod:
    fsGroup: 1000
  container:
    runAsUser: 1000
    runAsGroup: 1000
```

**Why 1000:** it is the standard first-user UID on most Linux distros, well above the system range, and consistent with the rest of the cluster.

If the cluster's PSS is upgraded to enforce the full `restricted` profile (seccomp `RuntimeDefault`, `allowPrivilegeEscalation: false`, `capabilities.drop: ["ALL"]`), those fields will need to be added to `pod.securityContext.container` in `values.yaml`.

## Configuration layout

The entire Authelia deployment is configured from a **single** values file. The chart's `templates/configMap.yaml` consumes the `configMap.*` structured values and renders them into a ConfigMap named `authelia` in the `authelia` namespace.

| File | Role |
|---|---|
| `security/authelia/helm/values.yaml` | Single source of truth: chart config + Authelia config + secret declarations |
| `terraform/templates/security/authelia/*.tftpl` | ExternalSecret templates (rendered to `security/authelia/helm/resources/`) |
| `terraform/vault-secrets/main.tf` | One-time secrets generation (run at cluster bootstrap) |
| Init container (in `values.yaml`) | Reads pre-hashed password, writes `/shared/users_database.yml` |

**Do not** add the users file to `configMap.extraConfigs` in `values.yaml`. The `extraConfigs` field appends paths to `X_AUTHELIA_CONFIG`, which makes Authelia parse those files as merged **configuration** at startup. The users file is not valid main-config YAML and Authelia will refuse to start. The init-container-generated file is mounted directly and read at runtime by the file backend.
