# Authelia

Authelia is the SSO / forward-authentication provider for the cluster, replacing Authentik. It is deployed at `auth.drmarchent.com` (taking over the hostname formerly used by Authentik).

## Architecture

A single ArgoCD Application (`authelia-helm`, sync-wave `-394`) deploys everything using multi-source:
- **Source 1:** Helm chart from `https://charts.authelia.com` (chart `authelia`, v0.11.6)
- **Source 2:** Values from `security/authelia/helm/values.yaml` — the single source of truth for the entire Authelia deployment (chart-level config + full `configuration.yaml` rendered as structured values, plus the `secret.additionalSecrets` declarations that tell the chart which Secrets to mount and where)
- **Source 3:** Raw manifests from `security/authelia/helm/resources/` — only the legacy mittwald `kubernetes-secret-generator` placeholder Secrets (see [follow-up cleanup](#follow-up-cleanup) below)

The configuration ConfigMap (`authelia`, in the `authelia` namespace) is rendered by the chart's `templates/configMap.yaml` from the `configMap.*` values — it is **not** a hand-maintained manifest. The chart also mounts four backing Secrets (`authelia-session-secret`, `authelia-storage-encryption-key`, `authelia-oidc-hmac-secret`, `authelia-grafana-client-secret`) via `secret.additionalSecrets` and sets the corresponding `*_FILE` env vars, so Authelia reads the secret material from the mounted files at startup.

The four Secrets are auto-generated in-cluster by the **External Secrets Operator** with `Password` generators (see `security/authelia/resources/*-externalsecret.yaml` and `*-generator.yaml`); the values flow into Vault via `PushSecret` and are consumed by Grafana via a separate `ExternalSecret` in the `monitoring` namespace.

```
Browser ──HTTPS──> Cilium Gateway (Envoy) ──> backend service (e.g. Harbor, Grafana)
                          │
                          ├── (per HTTPRoute filter) sub-request ──> Authelia :9091/api/authz/ext-authz/
                          │     - 200: allow, inject Remote-User/Remote-Groups headers
                          │     - 401/403: deny, redirect browser to https://auth.drmarchent.com
                          │     - The endpoint uses implementation=ExtAuthz (Envoy ext_authz protocol)
                          │       with authn_strategies CookieSession + HeaderAuthorization Basic
                          │
Browser ◀──redirect to login── Authelia (login UI at auth.drmarchent.com)
   │
   └── single admin user (file backend, password auto-generated by ESO Password generator)
```

## Deployment

```bash
# 1. ArgoCD picks up the change and deploys automatically
kubectl get applications -n argocd -w
# 3. Verify the pod is up
kubectl get pods -n authelia -w
# 4. Verify the certificate is issued
kubectl get certificate authelia-homelab-local -n authelia
# 5. Retrieve the auto-generated admin password
kubectl get secret authelia-admin-password -n authelia -o jsonpath='{.data.password}' | base64 -d
# 6. Browse https://auth.drmarchent.com
```

## OIDC secrets

Two Secrets back the OIDC provider. Both are auto-generated in-cluster by the External Secrets Operator's `Password` generator CRDs (see `security/authelia/resources/password-generator-*.yaml`).

| Secret | Key | Mounted at (in pod) | Purpose |
|---|---|---|---|
| `authelia/authelia-grafana-client-secret` | `client-secret` | `/secrets/authelia-grafana-client-secret/client-secret` | OIDC `client_secret` Grafana presents during the token exchange |
| `authelia/authelia-oidc-hmac-secret` | `hmac-secret` | `/secrets/authelia-oidc-hmac-secret/hmac-secret` | HMAC key for signing OIDC ID tokens |

The `authelia-grafana-client-secret` Secret is mounted as a directory at `/secrets/authelia-grafana-client-secret/` via the chart's `secret.additionalSecrets` mechanism; the configMap references the file at `client-secret` inside that directory using a `{{ secret "..." }}` template directive that Authelia resolves at startup (the chart sets `X_AUTHELIA_CONFIG_FILTERS=template`).

The `authelia-oidc-hmac-secret` is mounted the same way; the chart sets `AUTHELIA_IDENTITY_PROVIDERS_OIDC_HMAC_SECRET_FILE` to point at the file inside that directory, so Authelia reads it as `identity_providers.oidc.hmac_secret`.

Grafana reads the client secret from `monitoring/grafana-oidc-client-secret`, which is a separate `ExternalSecret` in the `monitoring` namespace that pulls from Vault at `authelia/grafana-oidc`. The Authelia-side `PushSecret` (see `security/authelia/resources/grafana-client-secret-pushsecret.yaml`) writes the value into Vault whenever the ESO-managed Secret changes. There is no copy-paste of values between manifests.

### Rotation

```bash
# OIDC client secret (Grafana picks up the new value on its next pod start)
# ESO Password generators are one-shot — to rotate, delete and re-apply the
# generator (which produces a new value) then restart the consumers.
kubectl delete generator authelia-grafana-client-secret-generator -n authelia
kubectl apply -f security/authelia/resources/password-generator-grafana.yaml
kubectl rollout restart deploy/authelia -n authelia
kubectl rollout restart deploy/grafana-deployment -n monitoring  # adjust to actual Grafana Deployment name

# OIDC HMAC signing key (rotating this invalidates all in-flight OIDC
# sessions — users will need to re-authenticate)
kubectl delete generator authelia-oidc-hmac-secret-generator -n authelia
kubectl apply -f security/authelia/resources/password-generator-hmac.yaml
kubectl rollout restart deploy/authelia -n authelia
```

## User Management

Authelia uses a single admin user. The password is auto-generated by the External Secrets Operator's `Password` generator (`security/authelia/resources/password-generator-admin.yaml`) and stored in the `authelia-admin-password` Secret (data key `password`). At pod startup, an init container:

1. Reads the plaintext password from the `authelia-admin-password` Secret
2. Hashes it with `authelia crypto hash generate argon2`
3. Writes `/shared/users_database.yml` containing the admin user entry

The users database file is mounted into the Authelia container at `/users_database.yml` via a shared `emptyDir` volume.

### Password rotation

To rotate the admin password:

1. Force ESO to regenerate the secret value (delete and re-apply the generator):
   ```bash
   kubectl delete generator authelia-admin-password-generator -n authelia
   kubectl apply -f security/authelia/resources/password-generator-admin.yaml
   ```
2. Restart the Authelia pod so the init container re-generates the users database: `kubectl rollout restart deploy/authelia -n authelia`
3. Retrieve the new password: `kubectl get secret authelia-admin-password -n authelia -o jsonpath='{.data.password}' | base64 -d`

### Adding a new user

Users can be added by editing the init container script in `security/authelia/helm/values.yaml`. Add additional user entries under the `users:` key in the `cat > /shared/users_database.yml` heredoc. Each user needs a `displayname`, `password` (hashed), `email`, and `groups` entry.

To generate a password hash for a new user:

```bash
docker run --rm authelia/authelia:latest \
  authelia crypto hash generate argon2 --password "<password>"
```

Then add the user to the init container script and update the `access_control` rules in `security/authelia/helm/values.yaml` (`configMap.access_control.rules`) as needed.

## How to add a new protected service

To put a new HTTPRoute behind Authelia, edit its `http-route/<service>.yaml` and add a `type: ExternalAuth` filter. Use the Harbor or Grafana HTTPRoute as a template.

You also need a `ReferenceGrant` in the `authelia` namespace allowing the new service's HTTPRoute to backendRef the `authelia` Service. Example for a service in the `myapp` namespace:

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: myapp-external-auth-authelia
  namespace: authelia
spec:
  from:
    - group: gateway.networking.k8s.io
      kind: HTTPRoute
      namespace: myapp
  to:
    - group: ""
      kind: Service
      name: authelia
```

Since there is only the admin user, any authenticated user has access to all protected services. If you add per-service groups later, add a corresponding `access_control` rule in `security/authelia/helm/values.yaml` (`configMap.access_control.rules`).

## Security context (non-root)

The cluster's `authelia` namespace is governed by the Kubernetes **Pod Security Standard `restricted`** admission policy, which rejects any pod that runs as UID/GID 0.

The chart's defaults for `pod.securityContext` are empty, so we set them explicitly in `security/authelia/helm/values.yaml:23-28`:

```yaml
securityContext:
  pod:
    fsGroup: 1000
  container:
    runAsUser: 1000
    runAsGroup: 1000
```

**Why 1000:** it is the standard first-user UID on most Linux distros, well above the system range, and consistent with the rest of the cluster (e.g. `storage/cloudnative-pg/operator/values.yaml` uses 1000 for the same reason — picking a high, fixed UID/GID avoids collisions).

**How it works with the Authelia image:** the upstream `entrypoint.sh` is designed to be run as root — it does `chown -R $PUID:$PGID /config` and `su-exec $PUID:$PGID authelia` so the SQLite DB at `/config/db.sqlite3` ends up owned by the configured user. When the container is started with `runAsUser: 1000`, the entrypoint detects non-root, skips `chown`+`su-exec`, and `exec authelia` directly. The `fsGroup: 1000` on the pod security context makes kubelet chown the Longhorn PVC to GID 1000 on first mount, so the new SQLite file is created with correct ownership.

If the cluster's PSS is upgraded to enforce the full `restricted` profile (seccomp `RuntimeDefault`, `allowPrivilegeEscalation: false`, `capabilities.drop: ["ALL"]`), those fields will need to be added to `pod.securityContext.container` in `values.yaml`.

## Configuration layout (the gotcha)

The entire Authelia deployment is configured from a **single** values file. The chart's `templates/configMap.yaml` consumes the `configMap.*` structured values and renders them into a ConfigMap named `authelia` in the `authelia` namespace. The init container in `pod.initContainers` generates the users database at startup.

| File | Role |
|---|---|
| `security/authelia/helm/values.yaml` | **Single source of truth.** Chart-level structural config (Deployment kind, PVC, service, security context, init container) + the full Authelia `configuration.yaml` expressed as structured values under `configMap.*` (server, session, storage, access_control, notifier, OIDC, …) + the `secret.additionalSecrets` declarations that tell the chart which Secrets to mount and via which env vars |
| Init container (in `values.yaml`) | Generates `/shared/users_database.yml` with the admin user entry by hashing the auto-generated password from the `authelia-admin-password` Secret. Mounted into the main container at `/users_database.yml` via a shared `emptyDir`. |

How the secrets flow:

1. ESO's `Password` generator (`security/authelia/resources/password-generator-<name>.yaml`) produces the secret value once at creation.
2. The `ExternalSecret` (`*-externalsecret.yaml`) references the generator and creates a Kubernetes Secret in the `authelia` namespace.
3. The chart's `secret.additionalSecrets` entry (in `values.yaml`) tells the chart to mount that Secret as a directory at `/secrets/<secret_name>/` in the pod.
4. The chart sets the appropriate `*_FILE` env var (`AUTHELIA_SESSION_SECRET_FILE`, `AUTHELIA_STORAGE_ENCRYPTION_KEY_FILE`, `AUTHELIA_IDENTITY_PROVIDERS_OIDC_HMAC_SECRET_FILE`, `AUTHELIA_IDENTITY_VALIDATION_RESET_PASSWORD_JWT_SECRET_FILE`) to point at the file inside that directory.
5. Authelia reads the file via the env var override at startup. The OIDC `client_secret` is the one exception: the chart renders it as a `{{ secret "..." }}` template directive in the configMap (e.g. `client_secret: '{{ secret "/secrets/authelia-grafana-client-secret/client-secret" }}'`), and the chart's `X_AUTHELIA_CONFIG_FILTERS=template` env var causes Authelia to resolve those directives at startup.

**Do not** add the users file to `configMap.extraConfigs` in `values.yaml`. The `extraConfigs` field appends paths to `X_AUTHELIA_CONFIG`, which makes Authelia parse those files as merged **configuration** at startup. The users file is not valid main-config YAML (the `users:` key has no meaning at the top level of the configuration schema) and Authelia will refuse to start with errors like `configuration key not expected: users.admin.displayname`. The init-container-generated file is mounted directly and read at runtime by the file backend.

## Follow-up cleanup

The directory `security/authelia/helm/resources/` contains five legacy `mittwald` `kubernetes-secret-generator` placeholder manifests (`admin-password-secret.yaml`, `oidc-client-secrets-secret.yaml`, `oidc-hmac-secret-secret.yaml`, `session-secret-secret.yaml`, `storage-encryption-key-secret.yaml`) that are no longer used. The active secret workflow is the External Secrets Operator (see [Architecture](#architecture)); these placeholders are dead code from an earlier approach. They should be deleted in a follow-up commit. Note in particular that `oidc-client-secrets-secret.yaml` uses a *different* Secret name (`authelia-oidc-client-secrets`) than the ESO-managed one (`authelia-grafana-client-secret`), which is harmless because nothing references the mittwald name.
