# Authelia

Authelia is the SSO / forward-authentication provider for the cluster, replacing Authentik. It is deployed at `auth.drmarchent.com` (taking over the hostname formerly used by Authentik).

## Architecture

A single ArgoCD Application (`authelia-helm`, sync-wave `-394`) deploys everything using multi-source:
- **Source 1:** Helm chart from `https://charts.authelia.com` (chart `authelia`, v0.11.6)
- **Source 2:** Values from `security/authelia/helm/values.yaml`
- **Source 3:** Raw manifests from `security/authelia/helm/resources/`

Raw manifests include the configuration ConfigMap, secrets (session, storage encryption, admin password), and an `ExternalSecret` for the Grafana OIDC client secret (used in Phase 2).

```
Browser ──HTTPS──> Cilium Gateway (Envoy) ──> backend service (e.g. Harbor, Grafana)
                          │
                          ├── (per HTTPRoute filter) sub-request ──> Authelia :9091/api/authz/ext-authz/
                          │     - 200: allow, inject Remote-User/Remote-Groups headers
                          │     - 401/403: deny, redirect browser to https://auth.drmarchent.com
                          │     - The endpoint uses implementation=ExtAuthz (Envoy ext_authz protocol)
                          │       with authn_strategies CookieSession + HeaderAuthorization Basic
                          │
Browser ◀──redirect to login── Authelia (login UI at auth.drmarchent.com)
   │
   └── single admin user (file backend, password auto-generated by mittwald secret-generator)
```

## Deployment

```bash
# 1. Push the secrets to Vault (see Vault Setup below)
# 2. ArgoCD picks up the change and deploys automatically
kubectl get applications -n argocd -w
# 3. Verify the pod is up
kubectl get pods -n authelia -w
# 4. Verify the certificate is issued
kubectl get certificate authelia-homelab-local -n authelia
# 5. Retrieve the auto-generated admin password
kubectl get secret authelia-admin-password -n authelia -o jsonpath='{.data.password}' | base64 -d
# 6. Browse https://auth.drmarchent.com
```

## Vault Setup

This cluster uses Vault to store secret API keys to avoid exposing them in manifests. The `ExternalSecret` in `helm/resources/external-secret.yaml` reads from this path:

- `kubernetes-homelab/authelia/grafana-oidc` — Grafana OIDC client_secret

For Phase 1 you can skip Vault entirely. The `oidc-client-secrets-secret.yaml` will be created empty and that's fine — the configuration doesn't reference it yet.

For Phase 2 (when we add OIDC for Grafana), run:

```bash
kubectl exec -ti vault-0 -n vault -- /bin/sh
vault login

# Grafana OIDC client secret
GRAFANA_SECRET=$(openssl rand -hex 32)
vault kv put kubernetes-homelab/authelia/grafana-oidc \
  client_secret="$GRAFANA_SECRET"
# IMPORTANT: copy $GRAFANA_SECRET — you'll paste it into
# monitoring/grafana/resources/grafana.yaml as the OIDC client_secret
```

## User Management

Authelia uses a single admin user. The password is auto-generated by the mittwald `kubernetes-secret-generator` operator and stored in the `authelia-admin-password` Secret. At pod startup, an init container:

1. Reads the plaintext password from the `authelia-admin-password` Secret
2. Hashes it with `authelia crypto hash generate argon2`
3. Writes `/shared/users_database.yml` containing the admin user entry

The users database file is mounted into the Authelia container at `/users_database.yml` via a shared `emptyDir` volume.

### Password rotation

To rotate the admin password:

1. Delete the Secret: `kubectl delete secret authelia-admin-password -n authelia`
2. The mittwald operator will recreate it with a new random password
3. Restart the Authelia pod so the init container re-generates the users database: `kubectl rollout restart deploy/authelia -n authelia`
4. Retrieve the new password: `kubectl get secret authelia-admin-password -n authelia -o jsonpath='{.data.password}' | base64 -d`

### Adding a new user

Users can be added by editing the init container script in `security/authelia/helm/values.yaml`. Add additional user entries under the `users:` key in the `cat > /shared/users_database.yml` heredoc. Each user needs a `displayname`, `password` (hashed), `email`, and `groups` entry.

To generate a password hash for a new user:

```bash
docker run --rm authelia/authelia:latest \
  authelia crypto hash generate argon2 --password "<password>"
```

Then add the user to the init container script and update the `access_control` rules in `security/authelia/helm/resources/configuration-configmap.yaml` as needed.

## How to add a new protected service

To put a new HTTPRoute behind Authelia, edit its `http-route/<service>.yaml` and add a `type: ExternalAuth` filter. Use the Harbor or Grafana HTTPRoute as a template.

You also need a `ReferenceGrant` in the `authelia` namespace allowing the new service's HTTPRoute to backendRef the `authelia` Service. Example for a service in the `myapp` namespace:

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: myapp-external-auth-authelia
  namespace: authelia
spec:
  from:
    - group: gateway.networking.k8s.io
      kind: HTTPRoute
      namespace: myapp
  to:
    - group: ""
      kind: Service
      name: authelia
```

Since there is only the admin user, any authenticated user has access to all protected services. If you add per-service groups later, add a corresponding `access_control` rule in `security/authelia/helm/resources/configuration-configmap.yaml`.

## Security context (non-root)

The cluster's `authelia` namespace is governed by the Kubernetes **Pod Security Standard `restricted`** admission policy, which rejects any pod that runs as UID/GID 0.

The chart's defaults for `pod.securityContext` are empty, so we set them explicitly in `security/authelia/helm/values.yaml:23-28`:

```yaml
securityContext:
  pod:
    fsGroup: 1000
  container:
    runAsUser: 1000
    runAsGroup: 1000
```

**Why 1000:** it is the standard first-user UID on most Linux distros, well above the system range, and consistent with the rest of the cluster (e.g. `storage/cloudnative-pg/operator/values.yaml` uses 1000 for the same reason — picking a high, fixed UID/GID avoids collisions).

**How it works with the Authelia image:** the upstream `entrypoint.sh` is designed to be run as root — it does `chown -R $PUID:$PGID /config` and `su-exec $PUID:$PGID authelia` so the SQLite DB at `/config/db.sqlite3` ends up owned by the configured user. When the container is started with `runAsUser: 1000`, the entrypoint detects non-root, skips `chown`+`su-exec`, and `exec authelia` directly. The `fsGroup: 1000` on the pod security context makes kubelet chown the Longhorn PVC to GID 1000 on first mount, so the new SQLite file is created with correct ownership.

If the cluster's PSS is upgraded to enforce the full `restricted` profile (seccomp `RuntimeDefault`, `allowPrivilegeEscalation: false`, `capabilities.drop: ["ALL"]`), those fields will need to be added to `pod.securityContext.container` in `values.yaml`.

## Configuration layout (the gotcha)

The Helm values, the configuration ConfigMap, and the init container each play a distinct role. Mixing them up is the most common source of `Authelia will not start` errors:

| File | Role |
|---|---|
| `security/authelia/helm/values.yaml` | Chart-level structural config (Deployment kind, PVC, service, security context, secret mounts, init container) |
| `security/authelia/helm/resources/configuration-configmap.yaml` | The actual Authelia `configuration.yaml` — server, session, storage, access_control, notifier, etc. |
| Init container (in `values.yaml`) | Generates `/shared/users_database.yml` with the admin user entry by hashing the auto-generated password from the `authelia-admin-password` Secret. |

**Do not** add the users file to `configMap.extraConfigs` in `values.yaml`. The `extraConfigs` field appends paths to `X_AUTHELIA_CONFIG`, which makes Authelia parse those files as merged **configuration** at startup. The users file is not valid main-config YAML (the `users:` key has no meaning at the top level of the configuration schema) and Authelia will refuse to start with errors like `configuration key not expected: users.admin.displayname`. The init-container-generated file is mounted directly and read at runtime by the file backend.
